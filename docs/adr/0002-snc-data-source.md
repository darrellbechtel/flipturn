# ADR 0002 — SNC data source: format, ethics, and fetch approach

**Status:** Accepted
**Date:** 2026-05-04
**Deciders:** Darrell Bechtel
**Spec link:** [`docs/superpowers/specs/2026-05-04-01-flipturn-mvp-design.md`](../superpowers/specs/2026-05-04-01-flipturn-mvp-design.md)

## Context

Plan 2 (workers) needs to fetch athlete and meet data from Swimming Canada's
public results infrastructure, the Tier-4 canonical archive of SNC-sanctioned
meet results. The exact data format determines the parser approach (cheerio for
HTML vs `JSON.parse` vs Playwright for JS-rendered SPAs) and constrains the rest
of the worker design.

The original plan referenced `results.swimming.ca` as the data source; the
spike found that SNC actually publishes athlete data through two cooperating
hosts (`www.swimming.ca` for athlete and meet metadata, `results.swimming.ca`
for live SPLASH Meet Manager output during recent meets), and we need to use
both selectively.

## Findings (from spike)

### Hosts and URL structure

- **Primary host (athlete pages, persistent meet records):** `https://www.swimming.ca/`
  - Athlete profile / personal bests: `https://www.swimming.ca/swimmer/<numeric_id>/`
    (e.g. `/swimmer/4030816/` for Ryan Cochrane)
  - Meet record page: `https://www.swimming.ca/swim-meet/<numeric_id>/`
    (e.g. `/swim-meet/8314/` for the 2010 Canadian Championships)
  - Individual result detail: `https://www.swimming.ca/result/<numeric_id>/`
    — **disallowed by robots.txt** (see below)
  - Page also exposes WordPress slug aliases: `/swimmer/<slug>/` (e.g.
    `penelope-oleksiak`) which redirect to the numeric URL.
- **Live-meet host (in-meet SPLASH output):** `https://results.swimming.ca/`
  - Per-meet HTML index: `https://results.swimming.ca/<meet_slug>/`
    (e.g. `/2026_Speedo_Canadian_Swimming_Open/`) — server-rendered HTML
    table-of-contents listing every event/round.
  - Per-event result lists: linked PDFs in the form `ResultList_<eventid>.pdf`,
    `StartList_<eventid>.pdf`. **PDF only, no HTML alternative.**
  - The site root `https://results.swimming.ca/` redirects to the most recent
    meet via `<meta http-equiv="refresh">`.

### Format and rendering

- **`/swimmer/<id>/` (athlete):** **fully server-rendered HTML.** A clean
  `<table>` with columns `Event | Course | Time | Date | Meet | Status`
  containing one row per personal best across all events and courses. Every
  row links to `/result/<id>/`, `/swim-meet/<id>/`, and uses anchor query
  strings like `?competition=Style-1`. Cheerio is sufficient. Page is ~70KB,
  exposes the data without any XHR. The HTML comments mention the data is
  cached from `api.swimrankings.net/athlete/<id>/personalbests` server-side,
  so we get whatever swimrankings has when www.swimming.ca's cache last
  refreshed.
- **`/swim-meet/<id>/` (meet on www.swimming.ca):** **SPA shell — JS-rendered
  via XHR to swimrankings API.** The static HTML has empty `<h1></h1>`,
  `<h2></h2>`, and an empty `<ul class="details-list">`; data populates
  client-side. Cheerio cannot read content from this URL. _Not used by our
  parser._
- **`results.swimming.ca/<meet_slug>/` (SPLASH index):** **server-rendered HTML.**
  Generated by SPLASH Meet Manager 11 (Splash Software GmbH) — recognizable
  by `<meta name="generator" content="SPLASH Meet Manager 11 ...">`. Contains
  a table-of-contents with the meet header (name, course, location, dates) and
  one row per event/round/gender, each linking to a per-event PDF.
- **Per-event PDFs:** the only place actual heat-by-heat results live for
  meets that flow through SPLASH. Out of scope for the MVP parser — we'll
  consume the meet-index HTML for "what events ran" and derive athlete-level
  results from the `/swimmer/<id>/` page.

### robots.txt

`https://www.swimming.ca/robots.txt` (HTTP 200, `text/plain`):

```
User-agent: *
Content-Signal: search=yes,ai-train=no
Allow: /

User-agent: Amazonbot
Disallow: /
[... other LLM-training bots disallowed including ClaudeBot, GPTBot, etc. ...]

Disallow: /result/*
Disallow: /resultat/*
Disallow: /en/result/*
Disallow: /fr/result/*

User-agent: *
Disallow:

Sitemap: https://www.swimming.ca/sitemap_index.xml
```

Interpretation:

- Generic `User-agent: *` → `Allow: /` with explicit `Disallow: /result/*`
  (the per-result detail pages). All other paths — including
  `/swimmer/<id>/` and `/swim-meet/<id>/` — are allowed for FlipTurnBot.
- LLM-training bots (`ClaudeBot`, `GPTBot`, `Amazonbot`, `Google-Extended`,
  etc.) are blocked entirely — FlipTurnBot is not in any of these categories
  (it's a domain-specific scraper for direct user-requested data, not
  generic AI training).
- `Content-Signal: ai-train=no` is honored; we do not use scraped content for
  AI/ML training.

`https://results.swimming.ca/robots.txt` returns **404 Not Found** — no
robots policy is published for the live-meet host. We will treat it
conservatively as if `User-agent: *: Allow: /` and apply our own politeness
defaults.

### Rate limiting and other signals

- **Rate-limit headers:** none of the standard headers (`X-RateLimit-*`,
  `Retry-After`, `RateLimit-Limit`) are present in normal responses.
- **However**, during the spike a single burst of ~3 requests/sec to
  `www.swimming.ca` triggered a Cloudflare `HTTP 429 Too Many Requests` with
  `retry-after: 10` and `cache-control: private, max-age=0, no-store`. This
  confirms there is an active rate-limit (Cloudflare WAF rule, not advertised).
  The 5-second-per-host token bucket from the design spec §6.3 stays well
  under this threshold.
- **Auth wall:** none. `/swimmer/<id>/` and `/swim-meet/<id>/` are publicly
  accessible. There is a "REMS Login" link in the header for SNC member
  registration, but it gates only club-registration features, not results.
- **Captcha:** none observed. Cloudflare interstitial may appear during a
  rate-limit event but not during normal politeness-respecting fetches.
- **Geo-block:** none observed from a Canadian residential connection.
- **Cache headers:** `cache-control: max-age=1800` and `cf-cache-status` on
  www.swimming.ca pages — Cloudflare-cached. `Last-Modified` is present.
  We can use `If-Modified-Since` for revalidation.
- **Encoding:** `text/html; charset=UTF-8` on www.swimming.ca; SPLASH index
  is `text/html; charset=UTF-8` declared via meta tag, response Content-Type
  is `text/html`.

### Data quality observations

- The athlete PB table sometimes has blank `meetExternalId` cells when the
  underlying SwimRankings record points to a meet not mirrored on swimming.ca.
- Some PB rows have a `Status` column value of `"RELAY"` — this is a split
  from a relay leg rather than a stand-alone official race. Parser must
  preserve this distinction.
- Times are formatted as `MM:SS.cc` or `SS.cc` with no padding.
- Athlete gender is **not** rendered on `/swimmer/<id>/`. We will need to
  source it elsewhere (e.g. infer from event genders the athlete competed in,
  or from the original parent-supplied lookup that produced the SNC ID).

## Decision

**Fetch approach:** **cheerio-on-static-HTML** (no Playwright required).

Specifically:

- `/swimmer/<id>/` on `www.swimming.ca` is the primary athlete data source.
  Fetch with `undici`, parse with `cheerio`.
- `https://results.swimming.ca/<meet_slug>/` is the secondary "live meet
  index" source for in-meet event lists. Same approach: `undici` + `cheerio`.
- We do **not** fetch `/swim-meet/<id>/` (JS-rendered, no static data) and
  we do **not** fetch `/result/<id>/` (robots-disallowed). Both restrictions
  are satisfied because the athlete-page table already contains everything
  we need at MVP scope (event, course, time, date, meet name, meet ID).

**Reasons:**

- Both target endpoints serve clean, server-rendered HTML — no JS execution
  required, no API discovery, no headless browser overhead.
- Cheerio + undici keeps the worker ~10× cheaper per fetch than a Playwright-
  based pipeline (no Chromium memory / startup cost).
- The cheerio parsers will need clear schemas; the Plan 3 TDD plan already
  assumes static HTML golden fixtures, which is what we captured.
- The robots.txt allows our use cases; the disallowed `/result/*` path is
  not on our fetch path.
- Cloudflare WAF rate-limit is comfortably above our 1-req/5s budget.

**Politeness defaults (per design spec §6.3):**

- User-Agent: `FlipTurnBot/0.1 (+https://flipturn.ca/bot; contact@flipturn.ca)`
- Rate limit: 1 req / 5s per host (token bucket, host = registrable domain).
- Daily per-host budget: 500 req/day.
- Honor robots.txt; cache for 24h. Refuse to fetch any path matching a
  `Disallow:` directive for our user-agent.
- Use `If-Modified-Since` / `ETag` where supported (`Last-Modified` is
  reliably present on www.swimming.ca; absent on SPLASH index).
- Back off on `429` per `Retry-After`; minimum back-off 60s even if header
  is shorter, to be conservative against the WAF.
- Do not retain content for ML training; we honor the `ai-train=no` content
  signal in robots.txt.

## Alternatives considered

- **Playwright headless browser** — would let us fetch `/swim-meet/<id>/` on
  www.swimming.ca (which renders meet metadata client-side) or even drive the
  Records & Rankings UI. Adds ~10× per-fetch cost (Chromium + DOM construction)
  and a heavy dependency (~300MB) to every worker container. Deferred — the
  athlete page already provides the meet IDs and names we need; we don't need
  the meet page to be human-readable. Revisit only if the parser cannot
  reconstruct meet metadata from the athlete-page rows.

- **Direct calls to `api.swimrankings.net`** — the public www.swimming.ca page
  proxies this API server-side (the HTML even includes
  `<!-- Key sr_api_https-api-swimrankings-net-athlete-XXXX-personalbests -->`
  as a transparency comment). Hitting the API directly would skip the Cloudflare
  cache and give us cleaner JSON. Rejected for now: it's a third-party endpoint
  not under SNC's umbrella, no robots policy or Terms we've reviewed, and adds
  a second host dependency. Revisit if/when we need data fields not exposed in
  the rendered HTML (e.g. athlete gender).

- **Bulk meet-list traversal** — instead of per-athlete fetches, walk every
  meet on `results.swimming.ca` in a date range and ingest all results.
  Bandwidth scales with meets not users; would also force us to parse PDFs
  (only format SPLASH provides). Rejected for MVP: closed beta is 10–20
  athletes, per-athlete fetch is much simpler and PDF parsing is a separate
  rabbit hole.

- **PDF parsing of SPLASH `ResultList_<id>.pdf`** — the only authoritative
  source for heat-by-heat splits and final places at meets that flow through
  SPLASH. Rejected for MVP scope: the athlete-page table already provides
  PB-level data, which is enough for the v1 parent-facing UI. Plan to
  revisit at v1.1 if/when splits become a feature requirement.

- **SwimCloud as a fallback** — already excluded by the design spec's
  anti-goals (one-off identity lookups OK; bulk scraping is not).

## Consequences

- The parser (Plan 3) can be built with `undici` + `cheerio` only; no
  Playwright, no PDF parser, no browser automation. This keeps the workers
  package lightweight and CI-friendly.
- Two distinct cheerio parsers are required: one for the WordPress-themed
  athlete page table (`tbody > tr > td.tc`, with anchor-based meet IDs) and
  one for the SPLASH meet index (`<table>`-heavy, gender encoded via
  `class="leftText genderM"`/`genderF`, stroke from section title rows).
- Athlete gender is not in the HTML — the schema layer (Plan 1) needs to
  permit `gender` to be `null` initially, with an enrichment path (manual
  override, parent-supplied during identity confirmation, or future API
  call). This was already a known constraint in the design spec but the
  spike confirms there is no easy fix at the source layer.
- Splits, places, and DSQ status are not visible on the athlete-page table.
  The schema (Plan 1) already allows `splits: []` and `place: null`; the
  parser will populate accordingly.

## Risks (carried forward)

- **HTML structure change** — both the WordPress theme and the SPLASH 11
  template are stable, but a redesign of swimming.ca (last major one was
  ~2023) would break the parser. Mitigation: Plan 3's TDD golden fixtures
  catch this immediately; we monitor parser-failure rate as a Sentry alert.
- **Cloudflare WAF tightening** — SNC could reduce the 429 threshold at any
  time. Our 1-req/5s budget gives 12× headroom against the observed limit,
  but we should add a metric for `429` occurrences and back off aggressively
  if they appear.
- **`api.swimrankings.net` outage** — the athlete page is server-cached from
  this API; if SwimRankings goes down, swimming.ca will eventually serve
  stale HTML. We surface "last updated" indirectly (the meet dates in the
  PB rows are the freshest signal) but do not have a strong guarantee.
- **PDF gap** — heat splits, official places, and DSQ flags only live in
  SPLASH PDFs. If a beta user asks "did my kid place 3rd in the 100 free at
  X meet?" we can't answer at MVP. Acceptable for the closed beta scope; flag
  to user via UI copy.
- **`/result/<id>/` robots disallow** — if a future product requirement
  forces us to fetch the per-result detail page (for e.g. heat number,
  reaction time), we are blocked by robots and would need to negotiate with
  SNC directly or change the requirement.
